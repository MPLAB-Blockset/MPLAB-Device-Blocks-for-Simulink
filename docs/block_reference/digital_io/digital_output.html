MCHP_Digital_Output - Digital Output Master Block

DigitalOutput Block Icon

Description

The MCHP_Digital_Output block is a high-level composite subsystem block that provides digital output functionality with optional read-back capability. This master block automatically manages internal sub-blocks to provide a clean, user-friendly interface for controlling GPIO output pins.

Key Features:

  • Composite subsystem architecture - contains Digital_Output_Write (always) and Digital_Output_Read (optional)
  • Dynamic subsystem management - automatically adds/removes read-back capability
  • Unified parameter interface - configure all options from single block dialog
  • Parameter synchronization - automatically propagates settings to internal blocks
  • Optional 1-sample delayed feedback - read previous output state when READPREVIOUS enabled
  • Flexible input/output formats - individual booleans or packed uint16/uint32
  • Full electrical configuration - initial values and output modes (push-pull/open-drain)

Automatic Subsystem Updates

When READPREVIOUS parameter changes, the callback performs:

% Pseudo-code for subsystem management
if strcmp(READPREVIOUS, 'on')
    % Add Digital_Output_Read block
    add_block('mchp_lib/Digital_Output_Read', [blk '/Read']);

    % Add Outport for feedback
    add_block('simulink/Sinks/Out1', [blk '/Outport']);

    % Connect: Read → Outport
    add_line(blk, 'Read/1', 'Outport/1');
else
    % Remove Digital_Output_Read block
    delete_block([blk '/Read']);

    % Remove Outport
    delete_block([blk '/Outport']);
end

% Update signal ordering ($) for proper port arrangement
MCHP_OrderInOut(blk);

Inputs:  led1_cmd, led2_cmd, led3_cmd (boolean)
Outputs: led1_state, led2_state, led3_state (boolean, 1 sample delayed)

Result:
- Control 3 LEDs independently
- Read back actual output state for diagnostics
- Use feedback for closed-loop LED sequencing
- 1-sample delay acceptable for LED applications

Implementation Details

Internal Block Communication

The master block propagates parameters to sub-blocks using these mechanisms:

1. Parameter Synchronization

% In MCHP_Digital_Output_Callback.m
% Copy parameters to Write sub-block
write_block = [blk '/Digital_Output_Write'];
set_param(write_block, 'PORT', NewMask.Values.PORT);
set_param(write_block, 'PIN', NewMask.Values.PIN);
set_param(write_block, 'PACK', NewMask.Values.PACK);
set_param(write_block, 'INIT_VALUE', NewMask.Values.INIT_VALUE);
set_param(write_block, 'OUTPUT_PIN_MODE', NewMask.Values.OUTPUT_PIN_MODE);

% Copy parameters to Read sub-block (if exists)
if strcmp(NewMask.Values.READPREVIOUS, 'on')
    read_block = [blk '/Digital_Output_Read'];
    set_param(read_block, 'PORT', NewMask.Values.PORT);
    set_param(read_block, 'PIN', NewMask.Values.PIN);
    set_param(read_block, 'PACK', NewMask.Values.PACK);
end

2. Port Ordering ($)

% Signal ordering function ensures proper Inport/Outport arrangement
% Called after subsystem modifications
MCHP_OrderInOut(blk);

% This function:
% - Orders input ports by signal name ($)
% - Orders output ports by signal name ($)
% - Maintains consistent port indices across model updates

Register Configuration (Delegated to Sub-blocks)

The master block doesn't directly configure registers - it delegates to internal blocks:

  • Digital_Output_Write: Configures TRIS (output direction), LAT (output state), ODC (open-drain), and initial values
  • Digital_Output_Read: Reads LAT register to provide feedback

See Digital_Output_Write and Digital_Output_Read documentation for register-level details.

1-Sample Delay Explanation

Why 1-Sample Delay? The read-back output represents the LAT register state from the previous execution cycle, not the current cycle. This is inherent to the execution order:
  1. Current cycle: Write block updates LAT register with new values
  2. Current cycle: Read block reads LAT register (sees previous values)
  3. Next cycle: Read block output updates with current cycle's written values
This 1/z delay is acceptable for most applications (LED feedback, diagnostics, state machines) but should be considered in timing-critical feedback loops.

Troubleshooting

Common Issues and Solutions

Issue Possible Cause Solution
Read-back output not updating READPREVIOUS = 'off' Enable READPREVIOUS parameter to add output ports
Read-back shows wrong values Expecting instantaneous feedback Remember 1-sample delay - read-back is LAT(k-1)
Outputs don't match inputs Pin configured as peripheral or analog Verify pin configuration, check ANSEL/RP registers
Open-drain mode not working Older chip family ODC only available on 33C/33A/PIC32. Use external circuit for older chips.
Parameters not updating sub-blocks Subsystem library link issue Break library link, update parameters, test again
Port arrangement changed Signal ordering after modification Normal - MCHP_OrderInOut() reorders ports by name

Subsystem Troubleshooting

Warning: Do not manually modify internal blocks (Digital_Output_Write, Digital_Output_Read) within the MCHP_Digital_Output subsystem. Always configure the master block - it will update internal blocks automatically. Manual changes may be overwritten during initialization.

Performance Considerations

Execution Overhead

Configuration Execution Time Code Size Notes
Write only (READPREVIOUS=off) ~5-10 cycles 10-20 instructions Minimal overhead
Write + Read (READPREVIOUS=on) ~10-20 cycles 20-40 instructions Additional LAT read
Simultaneous write (atomic) ~15-25 cycles 30-50 instructions Read-modify-write sequence
Optimization Tip: For maximum performance, use READPREVIOUS='off' when feedback is not needed. This eliminates the LAT read operation and reduces code size.
